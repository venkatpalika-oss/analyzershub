Communication / DCS Alarm

Decision tree for “No update”, “Bad quality”, “Comms fail”, “OPC/Modbus error”.

Step 1 — Identify Where the Failure Is

  • Case A: Values update on analyzer locally, but not in DCS → network/protocol/mapping.
  • Case B: Values do not update locally either → analyzer/run problem (check main page sections).

Step 2 — Physical Network Checks

  • Check Ethernet cable seated properly
  • Check switch port LEDs (link/activity)
  • If using fiber/media converter: check power and link
No link lights = physical issue → replace cable/port, check switch configuration.

Step 3 — IP / Subnet / Gateway

Verify:
  • IP address correct
  • Subnet mask correct
  • Gateway correct (if crossing VLAN/subnets)
  • No duplicate IP on network
Duplicate IP can cause intermittent update and bad quality.

Step 4 — Protocol Layer (OPC / Modbus / Other)

Check:
  • Protocol service status = RUNNING
  • Correct port open (ask IT/Automation if firewall rules exist)
  • Correct slave ID/register map (Modbus)
  • Correct OPC endpoint and certificate/trust (OPC UA if used)
If protocol restarted, wait 1–2 minutes then re-check DCS update.

Step 5 — Tag Mapping / Scaling / Quality Flags

Most common “bad quality” causes:
  • Wrong tag mapping (points shifted)
  • Wrong scaling/engineering units (CV/Wobbe)
  • DCS expecting faster update than GC cycle
  • Quality flag mapped incorrectly (e.g., always bad)
Action:
  • Compare one value on analyzer screen with same tag in DCS
  • Verify update timestamp is changing each GC cycle
  • Confirm DCS scan time aligns with GC analysis time

Escalation (If Still Not Fixed)

Send to automation/IT/vendor:
  • Analyzer IP/subnet/gateway
  • Protocol type + port
  • DCS tag list / register map
  • Screenshots: analyzer values + DCS values + alarm text
  • Switch port ID and link status
If local updates are stable and DCS updates after mapping fix → problem solved.
